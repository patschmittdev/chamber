Copilot Runtime Version Pin & the make:sandbox Mismatch

This is the failure we hit almost every day: npm run make:sandbox (or any packaging path) aborts in the prePackage Copilot-runtime prep hook with a version-mismatch error, even though the SDK smoke test passes and the dev server runs fine. This doc explains why, and exactly which two files to edit to fix it.

Symptom: make:sandbox fails at prePackage

Copilot CLI smoke test output did not include 1.0.55.
Output: GitHub Copilot CLI 1.0.57-2

Fix: bump the pin in both manifests to the version the launcher actually resolves to, then full npm install in each. See The fix.

Where the versions live

The @github/copilot CLI pin is declared in two places. They must match:

FileRoleRead by
chamber-copilot-runtime/package.json Packaged runtime manifest. The pinned CLI + SDK pair that ships inside the app. readRequiredVersions() in scripts/prepare-copilot-runtime.js
package.json (root) Dev/CI dependency pin so node_modules matches the packaged runtime. npm / the dev server & smoke:sdk

The companion @github/copilot-sdk pin lives in the same two files and usually does not change when only the CLI build moves. Leave it alone unless the SDK itself is bumped.

Do not touch the unrelated chamber-copilot entry in the root package.json — that is a different package (Chamber's own, e.g. 0.5.11), not the GitHub Copilot CLI.

Why it happens

The npm pin does not control the runtime binary version on a developer machine. The chain:

  1. The @github/copilot npm package ships npm-loader.js, which spawns the platform binary @github/copilot-win32-x64/copilot.exe.
  2. That copilot.exe is a bootstrap launcher. It self-resolves to the newest build present in the global cache at ~/.copilot/pkg/win32-x64/ — ignoring the pinned npm version entirely.
  3. Every time you run copilot update (or the CLI auto-updates), a new build lands in that global cache (e.g. 1.0.55-1, 1.0.56-1, 1.0.57-2 all side-by-side).
  4. The prep hook's smokeTestRuntime() runs copilot.exe --version, gets the newest cached build, and compares it against the manifest pin. Mismatch → abort.
Why CI doesn't hit this: CI starts with a clean global cache, so the launcher resolves to exactly the pinned/installed build. Only developer machines with a populated ~/.copilot/pkg/ cache see the drift. That's why it bites locally "every day" but the pipeline stays green.

The fix

Align the pin with whatever the launcher resolves to. Find the resolved build:

# what the launcher actually runs:
copilot --version
# or inspect the cache:
Get-ChildItem $HOME\.copilot\pkg\win32-x64\

Then set both manifests to that exact version (e.g. 1.0.57-2):

  • chamber-copilot-runtime/package.json → "@github/copilot"
  • package.json (root) → "@github/copilot"

Refresh the lockfiles with a full npm install in each location:

npm install                       # root
cd chamber-copilot-runtime && npm install
Use full npm install, not npm install --package-lock-only. On some npm versions the lock-only flow strips optional cross-platform entries (e.g. @emnapi/core, @emnapi/runtime) that CI later requires, producing a different failure downstream.

Validate, in order:

npm run smoke:sdk      # SDK/CLI compatibility on the new pin
npm run make:sandbox   # expected version now == launcher-resolved version

Governance note

Bumping the @github/copilot / @github/copilot-sdk pair is a change to the committed runtime contract. Per copilot-instructions / AGENTS.md, a real bump warrants a coordinated CHANGELOG.md entry and validation via npm run smoke:sdk + npm run make:sandbox. A purely local re-alignment to stop the daily packaging abort can be done in isolation, but if you commit the pin change, treat it as a runtime-contract bump.
