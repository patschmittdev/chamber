Copilot Runtime Version Pin & Packaging Validation

Chamber ships an exact Copilot CLI and SDK pair. Packaging validates that committed runtime without consulting the developer's self-updated Copilot cache.

Invariant: package metadata and the isolated executable agree

@github/copilot-sdk package: 1.0.6-preview.1
@github/copilot package: 1.0.69-1
Isolated copilot --version: 1.0.69

A normal copilot --version may report a newer self-updated build. That user-cache version is not the packaged runtime pin.

Where the versions live

The @github/copilot CLI pin is declared in two places. They must match:

FileRoleRead by
chamber-copilot-runtime/package.json Packaged runtime manifest. The pinned CLI + SDK pair that ships inside the app. readRequiredVersions() in scripts/prepare-copilot-runtime.js
package.json (root) Dev/CI dependency pin so node_modules matches the packaged runtime. npm / the dev server & smoke:sdk

The companion @github/copilot-sdk pin lives in the same two files and usually does not change when only the CLI build moves. Leave it alone unless the SDK itself is bumped.

Do not touch the unrelated chamber-copilot entry in the root package.json — that is a different package (Chamber's own, e.g. 0.5.11), not the GitHub Copilot CLI.

Why normal and packaged version checks differ

The npm pin controls the bundled platform package, while a normal developer invocation may use cached updates:

  1. The @github/copilot npm package ships npm-loader.js, which spawns the platform binary @github/copilot-win32-x64/copilot.exe.
  2. The launcher may select the newest build in ~/.copilot/pkg/<platform-arch>/ for normal interactive use.
  3. Every time you run copilot update (or the CLI auto-updates), a new build lands in that global cache (e.g. 1.0.55-1, 1.0.56-1, 1.0.57-2 all side-by-side).
  4. The prep hook runs copilot --version with temporary HOME and USERPROFILE paths, so the global cache cannot affect validation.
Do not pin to the global cache version. A cached build may not exist as an npm release and is not the runtime Chamber will package.

Changing the packaged runtime

Choose an exact version that exists on npm, then update both manifests:

  • chamber-copilot-runtime/package.json → "@github/copilot"
  • package.json (root) → "@github/copilot"

Refresh both lockfiles with a full install:

npm install                       # root
cd chamber-copilot-runtime && npm install
Use full npm install, not npm install --package-lock-only. On some npm versions the lock-only flow strips optional cross-platform entries (e.g. @emnapi/core, @emnapi/runtime) that CI later requires, producing a different failure downstream.

Validation checks the isolated bundled executable, not the user's cached CLI:

npm run smoke:sdk      # SDK/CLI compatibility on the new pin
npm run make:sandbox   # isolated executable version == committed pin

Governance note

Bumping the @github/copilot / @github/copilot-sdk pair changes the committed runtime contract and requires a changelog entry plus npm run smoke:sdk and packaging validation. A developer's global cache never changes Chamber's committed runtime contract.
